# 会计准则内容整理 SOP

## 一、信息来源优先级

为确保准则内容的准确性和权威性，必须按照以下优先级使用信息来源：

### 优先级排序

| 优先级 | 来源 | 说明 |
|--------|------|------|
| **P0** | 财政部会计司 (kjs.mof.gov.cn) | 中国准则官方发布平台，准则原文最权威来源 |
| **P1** | 财政部会计准则委员会 (casc.org.cn) | 准则、解释、案例权威发布，包含完整修订历史 |
| **P2** | 四大官网 (普华永道/德勤/安永/毕马威) | 专家解读、实务要点 |
| **P3** | 国内事务所 (立信/天健/信永中和等) | 实务案例、技术通讯 |

## 二、准则内容整理标准流程

### Step 1: 确定准则信息

在整理任何准则前，必须确认以下信息：

- [ ] 准则编号和名称
- [ ] 所有版本发布日期和文号
- [ ] 各版本当前状态（现行有效/已废止/已修订）
- [ ] 生效日期

### Step 2: 获取原文

1. 访问财政部会计司或 casc.org.cn 获取准则原文
2. 如有修订，查找所有版本（2014年后多项准则修订）
3. 记录原文链接和应用指南链接到 metadata
4. 为每个版本创建独立文件

### Step 3: 内容核对

- [ ] 对照官方原文逐条核对
- [ ] 检查章节结构是否完整
- [ ] 确认条款编号和内容准确
- [ ] 特别关注有修订的准则（如第2、7、9、12号等）

### Step 4: 格式规范化

按照本 SOP 第三章的格式要求进行整理

### Step 5: 相关准则关联

在 related 字段中添加相关准则链接：
- 同类资产准则（如存货准则关联资产减值）
- 引用准则（如非货币性资产交换引用债务重组）
- 配套准则（如收入准则关联建造合同）

## 三、内容格式标准

### 3.1 文件命名规范

#### 单版本准则
```
{编号}-{准则简称}.md
```

示例：
- `00-基本准则.md`
- `01-存货.md`

#### 多版本准则
```
{编号}-{准则简称}-{版本年份}.md
```

示例：
- `02-长期股权投资-2006.md` （2006年版本）
- `02-长期股权投资-2014.md` （2014年修订版，现行有效）
- `07-非货币性资产交换-2006.md` （2006年版本，已废止）
- `07-非货币性资产交换-2019.md` （2019年修订版，现行有效）

### 3.2 Frontmatter 规范

```yaml
---
title: "企业会计准则第X号——XXX"
short_title: "XXX"
date: YYYY-MM-DD
category: "企业会计准则"
number: X
status: "现行有效" | "已废止" | "已修订"
summary: "一句话描述本准则的核心内容"
tags: ["标签1", "标签2", ...]
related:
  - title: "企业会计准则第X号——XXX"
    slug: "XX-XXX"
origin_url: "http://kjs.mof.gov.cn/zt/kjzzss/..."
application_guide:
  - title: "企业会计准则第X号——XXX应用指南"
    url: "http://kjs.mof.gov.cn/..."
versions:
  - title: "企业会计准则第X号——XXX（2006版）"
    slug: "XX-XXX-2006"
    status: "已废止"
  - title: "企业会计准则第X号——XXX（2014版）"
    slug: "XX-XXX-2014"
    status: "现行有效"
---
```

**字段说明：**

| 字段 | 必填 | 说明 |
|------|------|------|
| title | ✅ | 完整准则名称 |
| short_title | ✅ | 简短名称（用于导航） |
| date | ✅ | 发布日期（首次发布或最新修订） |
| category | ✅ | 固定值"企业会计准则" |
| number | ✅ | 准则编号（0-42） |
| status | ✅ | 现行有效/已废止/已修订 |
| summary | ✅ | 50字以内的准则简介 |
| tags | ✅ | 关键词标签（3-5个） |
| related | ✅ | 相关准则关联（2-4个） |
| origin_url | ✅ | 财政部会计司官方原文链接 |
| application_guide | ✅ | 应用指南链接（如有） |
| versions | ✅ | 其他版本链接（多版本准则必填） |

### 3.3 正文结构规范

```
# 准则标题

（发布信息：日期、文号，可选）

## 第一章 总则
（第一条-第X条）

## 第二章 XXX
（第一条-第X条）

## 第三章 XXX
...

## 实务要点（可选）
- 对比表格
- 会计分录示例
- 常见问题

## 相关准则
- [准则名称](/docs/01-企业会计准则/XX-XXX)
```

### 3.4 条款编号格式

- **章标题**：`## 第一章 总则`
- **条标题**：`**第一条** 条款内容`
- **款项**：`（一）款项内容`
- **项**：`1. 项目内容`

### 3.5 引用其他准则

当正文引用其他准则时，使用以下格式：

```markdown
适用《企业会计准则第X号——XXX》
```

### 3.6 禁止事项

- ❌ 禁止添加个人解读（可添加到"实务要点"章节并注明来源）
- ❌ 禁止修改官方原文措辞
- ❌ 禁止省略条款（即使看起来"不重要"）
- ❌ 禁止使用非官方简称

## 五、质量检查清单

完成每条准则整理后，必须逐项检查：

### 内容核对
- [ ] 原文逐条对照财政部会计司官网
- [ ] 章节结构完整（总则/确认/计量/披露等）
- [ ] 条款编号连续无误
- [ ] 引用其他准则的表述准确

### 格式检查
- [ ] Frontmatter 字段完整（含 origin_url、application_guide、versions）
- [ ] 文件命名符合规范
- [ ] 相关准则链接正确
- [ ] 多版本准则已分别保存

### 链接验证
- [ ] related 中链接的文件存在
- [ ] origin_url 官方链接有效
- [ ] application_guide 应用指南链接有效
- [ ] versions 中各版本链接存在

#### 脚本2：内容一致性检查脚本


**输出结果**：

### 6.2 常见问题与处理方案

#### 问题1：错误的通用链接

**症状**：origin_url 指向 `http://kjs.mof.gov.cn/zt/kjzzss/kuaijizhunzeshishi/`（目录页而非具体文件）

**影响**：无法直接访问准则原文

**解决方案**：
1. 从 JSON 数据源查找对应准则的具体 URL
2. 格式：`https://kjs.mof.gov.cn/zt/kjzzss/kuaijizhunzeshishi/YYYYMMDD/tYYYYMMDD_XXXXXX.htm`
3. 运行修复脚本自动更新

**已发现案例**：
- 31-现金流量表.md
- 32-中期财务报告.md  
- 34-每股收益.md
- 35-分部报告.md
- 37-金融工具列报-2006.md

#### 问题2：内容缺失

**症状**：md 文件正文长度远短于 JSON 数据源（如 300字符 vs 2500字符）

**影响**：准则条款不完整，无法正常使用

**解决方案**：
1. 运行内容一致性检查脚本，识别内容缺失文件
2. 使用 update_md_from_json.py 脚本用 JSON 完整内容更新
3. 保留原文件的 frontmatter 信息

**已发现案例**：
- 32-中期财务报告.md（307→2504字符）
- 33-合并财务报表.md（429→9345字符）
- 34-每股收益.md（344→1794字符）
- 35-分部报告.md（316→3138字符）
- 37-金融工具列报-2006.md（201→24753字符）

#### 问题3：版本匹配错误

**症状**：md 文件内容（2006版）与 JSON 匹配到了不同版本（2014版），导致相似度低

**影响**：内容对比时产生误报

**解决方案**：
1. 确认 md 文件的版本年份
2. 检查 origin_url 是否指向正确版本
3. JSON 中按 URL 精确匹配，而非标题模糊匹配
4. 相似度低的文件需人工复核，不一定是错误

### 6.4 更新流程建议

#### 新增准则流程
1. 从官方获取准则原文
2. 创建 md 文件并填写 frontmatter
3. **必须**：运行检查脚本验证
4. 提交前确认 origin_url 和内容完整性

#### 批量更新流程
1. 更新 JSON 数据源（重新抓取官方内容）
2. 运行内容一致性检查
3. 对警告文件使用更新脚本修复
4. 重新运行检查确认修复成功

#### 定期检查（每月）
1. 检查 origin_url 有效性
2. 检查内容完整性
3. 更新失效链接
4. 补充缺失内容

## 七、经验教训总结

### 本次检查发现的问题

| 问题类型 | 数量 | 原因分析 | 预防措施 |
|---------|------|---------|---------|
| 错误通用链接 | 5个 | origin_url 指向目录页而非具体文件 | 使用审计脚本定期检查 |
| 内容缺失 | 5个 | md 文件只包含概述，缺少完整条款 | 使用一致性检查脚本 |
| 版本匹配错误 | 11个 | JSON 中有多个版本，匹配到错误版本 | 使用 URL 精确匹配 |

### 关键发现

1. **JSON 数据源是最可靠的参考**：经过多次验证，JSON 中的内容完整且准确
2. **md 文件存在系统性内容缺失**：特别是32、33、34、35、37号准则
3. **origin_url 配置不规范**：存在使用目录页链接而非具体文件链接的情况
4. **相似度检查有局限性**：不同版本准则内容差异大，低相似度不一定是错误

### 改进建议

1. **建立自动化检查 CI**：每次提交前自动运行检查脚本
2. **维护 JSON 数据源作为单一真相源**：md 文件内容应与 JSON 保持一致
3. **规范化 URL 格式**：所有 origin_url 必须指向具体文件，禁止目录页链接
4. **定期审计机制**：每月运行全套检查脚本，生成审计报告
